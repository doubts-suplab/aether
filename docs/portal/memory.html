Team memory that strengthens together, shares privately

Memory generalises Core's per-user cognition to a team. Every distinct contributor who accesses a shared memory raises its salience. And when a memory is federated across instances, only a bounded summary crosses the boundary — never the team, the contributors, or the raw record.

Why the projection matters

  1. Privacy is the default; sharing is opt-in. Only FEDERATED memories in federation-enabled tenants are ever eligible. A PRIVATE or TENANT memory simply doesn't exist to a remote query.
  2. The boundary strips identity. The projection carries a bounded summary and coarse provenance only — no team_id, no contributor names, no raw IDs. You can see the two envelopes never match.
  3. Salience is collective. contributorCount rises only for distinct contributors — repeated access by the same teammate reinforces strength but doesn't inflate the count.
Bounded context: Memory owns shared/team memory. Personal memory stays in Core; documents stay in Vault. Every query is scoped by tenant_id and team_id — and federated results never carry either.
